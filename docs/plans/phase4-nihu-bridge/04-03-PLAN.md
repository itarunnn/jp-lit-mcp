<objective>
プラン 04-02 で実装した `createNihuBridgeAdapter` を `createServer` に追加し、`NIHU_BRIDGE_SEARCH_URL` / `NIHU_BRIDGE_RECORD_BASE_URL` 環境変数を読めるようにする。あわせて `searchService` の SearchInput 型を `nihu_bridge` filters に対応させる（`jp_lit_search` ツールの payload が adapter まで届くよう型を通す）。

Purpose: nihuBridge を実際に MCP ツール経由で叩ける状態にする。横断検索の既定対象には **入れない**（nihu_bridge は source 指定専用）。
Output: `npm run smoke:mcp` が tools 7 件公開を維持し、`source=nihu_bridge` の検索リクエストが server からfetch まで貫通する。
</objective>

<context>
- 既存の adapter 登録パターン（src/server.ts L221-236）に倣う。
- 環境変数の読み取り箇所（resolveAdapterOptionsFromEnv L130-214）に `nihuBridge: { searchUrl, recordBaseUrl }` を追加する。
- `searchService.ts` の `SearchInput.filters` 型に `nihu_bridge?: NihuBridgeSearchFilters` を追加する（`tools/jpLitSearch.ts` の handler は parsed.filters をそのまま渡すだけなので、ここの型を直すと一気に貫通する）。
- 横断検索の既定対象（`CROSS_SOURCE_ORDER`）には **追加しない**。`nihu_bridge` は source 指定専用にする（jdcat / irdb と同方針）。
- `jp_lit_get_record` ツールは server.ts の adapters 配列に adapter が登録されれば自動的にルーティングされる設計（`tools/jpLitGetRecord.ts` 内で `registry.get(source).getRecord(source_id)` を呼ぶ）。よって本プランで `createNihuBridgeAdapter` を adapters に追加することで `jp_lit_search` / `jp_lit_get_record` 両方の `source=nihu_bridge` ルーティングが同時に有効になる。
</context>
