# IRDB Source 実装計画

**Goal:** `jp_lit_search` / `jp_lit_get_record` に `source: "irdb"` を追加し、日本の機関リポジトリ由来の紀要・学位論文・研究報告書を探索できるようにする。

**Scope:**

- `irdb` source の追加
- search adapter
- record/detail adapter
- 共通スキーマへの正規化
- fixture / adapter test / API notes / README 更新

**Out of Scope:**

- PDF 本文取得
- PDF OCR / テキスト抽出
- 既定横断検索への投入
- OAI-PMH ハーベスト基盤の実装
- IRDB と CiNii / NDL の自動重複統合

## 方針

- 初版は `source=irdb` 指定専用で始め、既定横断検索には入れない。
- 検索は IRDB 公式 `OpenSearch` を使う。
- `OAI-PMH` はハーベスト用として認識し、対話的検索 API の代替とは見なさない。
- `get_record` は、まず `IRDB アイテム詳細画面` と `ソース元 URI` のどちらを主 detail とするか実調査してから決める。
- source 固有情報は `source_metadata` と `raw` に保持する。
- 重い取得は入れない。PDF 本体や本文抽出は別フェーズに切る。

## 公式確認済みの前提

- IRDB は `OpenSearch` に対応している。
- OpenSearch endpoint は `https://irdb.nii.ac.jp/opensearch/search`
- 主な検索パラメータ:
  - `q`
  - `title`
  - `author`
  - `keyword`
  - `journal`
  - `issn`
  - `publisher`
  - `fulltext`
  - `count`
  - `start`
  - `format` (`rss` / `atom`)
- OpenSearch レスポンスには少なくとも次が含まれる。
  - `title`
  - `link`（IRDB アイテム詳細画面 URL）
  - `author`
  - `subject`
  - `publisher`
  - `URI`（ソース元アイテム詳細画面 URL）
  - `irname`
  - `pubDate`

## 設計判断

- source 名は単純に `irdb` とする。
- `jp_lit_search` の `sort_by` / `sort_order` は初版では未対応とする。
- `fulltext=1` のような IRDB 固有 filter は、初版では MCP 公開引数に出さない。
- `get_record` で返す `url` は `IRDB 詳細画面 URL` を優先し、`source_metadata.source_uri` に原機関 URI を残す案を第一候補にする。
- 既定横断検索には入れない。
  - 理由: 紀要・学位論文・報告書・リポジトリ資料が広く混ざり、`cinii_articles` / `jstage_articles` よりノイズが強い可能性が高い。

### Task 0 実績メモ

- OpenSearch は `atom` を優先採用する。
  - `rss` より `language`, `contributor`, `prism:*`, `updated` などが豊富
- `count` は `20 / 50 / 100` のみ有効
  - MCP の `limit` とはそのまま一致しないため、adapter 側で補正が必要
- `get_record` は `IRDB 詳細画面 HTML` を主経路にする
  - 原機関 `URI` は `source_metadata.source_uri` に保持
  - 理由: 原機関側はメンテナンス中などで不安定な場合がある
- 詳細画面では次を安定取得できる見込み
  - タイトル
  - 作成者
  - 内容注記
  - 出版者
  - 日付
  - 言語
  - 資源タイプ
  - 出版タイプ
  - 識別子
  - 収録誌情報
  - ファイル URL / MIME type
  - 更新日時

## 想定リスク

- `get_record` の安定入口が弱い場合、初版は `search` より detail の実装コストが高い
- OpenSearch は検索に十分でも、detail が HTML 依存になる可能性がある
- 原機関 URI と IRDB 詳細 URL のどちらを正規 URL とするかで設計が揺れやすい
- 機関ごとに metadata の揺れが大きく、`summary` / `subjects` / `pdf_url` の欠落率が高い可能性がある
