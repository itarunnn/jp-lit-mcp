# Skill And MCP Trigger Rules

## 目的

`jp-lit-research` Skill と `ndl-jp-lit-mcp` の役割分担、および発火条件を明確化する。特に、曖昧な依頼でいきなり横断検索が走るのを避け、最小限の確認と最小限の source 選定を優先する。

## 基本原則

- `Skill` は「調査の判断役」
- `MCP` は「検索・取得の実行役」
- 曖昧な依頼では `Skill` を先に動かし、必要になるまで `MCP` を叩かない
- `MCP` を叩くときも、最初から source 未指定の横断検索を既定にしない
- `Skill` は単発検索用ではなく、試行錯誤を支える対話的な調査セッションの進行役として扱う
- `MCP` 単独利用では、理想的にはユーザーが source を決める

---

## 1. Skill ありのとき

### 1-0. Skill の位置づけ

`jp-lit-research` Skill は、1回検索して終わるための補助ではない。文献調査の過程を対話で前に進めるための進行役として扱う。

想定する流れ:

1. 依頼を受ける
2. 仮説を立てる
3. 最小限の source / query で試す
4. 結果を読む
5. 次の一手を決める
6. 必要なら query や source を変えて再検索する
7. 途中経過を整理して返す

つまり Skill の役割は、最初の query を作ることだけではなく、検索結果を受けて次の query や次の source を決め続けることにある。

### 1-1. Skill の発火条件

次のような依頼では、`jp-lit-research` Skill を広めに発火させる。

- 文献を探したい
- 論文を探したい
- 本や雑誌を探したい
- 所蔵を調べたい
- 初出を調べたい
- デジコレで本文を探したい
- 図版や挿絵を探したい
- 古い表記や旧字で探したい
- どの DB を使えばいいか分からない

### 1-2. Skill は発火するが、すぐに MCP は叩かない条件

次のような依頼では、まず意図判定や確認を行い、`MCP` を遅延発火する。

- `ちょっと調べて`
- `ざっと調べて`
- `何で調べればいい？`
- `どのDBが向いてる？`
- `こういうテーマって見つかる？`
- 時代・資料種別・テーマ範囲が曖昧

### 1-3. Skill ありで MCP を叩く条件

次のいずれかが満たされたら `MCP` を実行してよい。

- ユーザーが検索実行を明示している
- query がある程度具体化した
- source が 1〜2 個に絞れた
- `研究ガイド` ではなく実検索フェーズに入った

### 1-4. Skill ありでの初手

- いきなり source 未指定の横断検索はしない
- まずは 1 問だけ確認するか、最小 source 1〜2 個へ絞る
- `quick` でも、最初から広い横断検索を既定にしない

例:

- `ちょっと調べて`
  - 検索しない
  - まず「論文中心か、本や所蔵中心か、デジコレ本文中心か」を確認する
- `ちょっと調べて。論文中心で`
  - `cinii_articles` / `jstage_articles` から始める
- `ちょっと調べて。所蔵も知りたい`
  - `ndl_catalog` / `cinii_books` から始める
- `ちょっと調べて。デジコレ本文も見たい`
  - `ndl_digital` と OCR 系を優先する

### 1-5. Skill ありでの再探索ルール

1回の検索結果を受けて、次の一手を決める。

- ヒット 0 件
  - 表記ゆれ、旧字、上位語、下位語へ展開する
  - source を変える
- ヒットが多すぎる
  - 年代、資料種別、主題で絞る
  - source を狭める
- 有力な 1 件が見つかった
  - 書誌詳細を見る
  - 著者名、掲載誌名、件名から次 query を作る
  - 別 source で追跡する
- `ndl_digital` で見つかった
  - OCR、ページ特定、図版検索へ進むか判断する

この再探索ループを支えるのが Skill の中心的な役割である。

## 2. MCP 単体のとき

### 2-0. MCP 単体の位置づけ

`MCP` 単体は、対話的に調査方針を組み立てるためのものではなく、検索や取得を実行するための道具として扱う。

そのため、理想的にはユーザーが次をある程度決めている状態を前提にする。

- どの source を使うか
- 何を探すか
- 何をしたいか

`Skill` なしでも source 未指定の横断検索は可能だが、これは主役ではなく補助的な入口として扱う。

### 2-1. MCP 単体で発火する条件

次の条件がそろっていれば `MCP` を発火してよい。

- ユーザーが `探して` / `検索して` / `調べて` と言っている
- 対象がある程度具体的
  - 人名
  - 書名
  - 作品名
  - テーマ
  - キーワード
- 欲しい資料種別がある程度見える
  - 論文
  - 本
  - 所蔵
  - 会議録
  - デジコレ本文
  - 図版

### 2-2. MCP 単体ですぐには発火しない条件

- `何で調べればいい？`
- `どのDBが向いてる？`
- `ちょっと調べて` だけで対象が曖昧
- 文献調査依頼なのか一般説明なのか不明

### 2-3. MCP 単体の初手

- 最初から source 未指定の横断検索を既定にしない
- 最小 source 1〜2 個から始める
- 推奨は source 明示

初手の目安:

- 論文っぽい
  - `cinii_articles`
  - `jstage_articles`
- 本・所蔵っぽい
  - `ndl_catalog`
  - `cinii_books`
- デジコレ資料っぽい
  - `ndl_digital`
- 会議録っぽい
  - `kokkai_minutes`
  - `teikoku_minutes`
- 文化資源・図版っぽい
  - `japan_search`
  - 必要なら `jp_lit_search_illustrations`

## 3. source 未指定の横断検索の扱い

source 未指定の横断検索は、次の場合にだけ使う。

- ユーザーがすでに実検索を求めている
- query が十分具体的
- 論文・本・デジコレのどれかに寄せる理由が特にない
- まず広く候補を見てから絞る方が合理的

逆に、次の場合は使わない。

- `ちょっと調べて`
- query が短すぎる
- 資料種別が曖昧
- まず DB を選ぶべき段階

---

## 4. 運用ルールとしての結論

- `Skill` は広めに発火してよい
- `MCP` は遅延発火でよい
- `ちょっと調べて` では横断検索しない
- `quick` は「横断検索1回」ではなく「最小限の確認または最小限の source 検索」と定義する
- source 未指定の横断検索は、曖昧依頼の初手には使わない
- `Skill` は対話的な再探索ループを回すためのもの
- `MCP` 単独では、source をユーザーが決める使い方を推奨する
