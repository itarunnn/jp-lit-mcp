# CRD Integration Positioning Memo

## 前提

レファレンス協同データベース（レファ協、CRD）には公式 API があり、技術的には `ndl-jp-lit-mcp` へ統合可能である。

しかし、今の `ndl-jp-lit-mcp` の中心は「書誌・所蔵・論文・OCR・図版などの検索と取得」であり、レファ協の価値はそれとは少し異なる。

レファ協の主な価値:

- 類似の調査事例
- 調べ方マニュアル
- 回答に至るプロセス
- 参考資料の手がかり

これは単なる書誌検索 source というより、**調査支援・探索支援のための source** に近い。

---

## 結論

レファ協は、今の `jp_lit_search` に通常の source として雑に追加しない方がよい。

理由:

1. 書誌・論文・会議録・OCR と性格が違う
2. `jp_lit_search` の結果一覧に「文献」と「調べ方ガイド」が混ざると分かりにくい
3. 今の Skill 設計では、調べ方支援の主役は Skill 側にある

したがって、レファ協は **research_guide intent を補強する補助 source** として位置づけるのが自然である。

---

## どういう役割で入れるか

### 適した役割

- 調査の入口を示す
- 似た質問の調査プロセスを参照する
- 参考資料や索引類の手がかりを得る
- 自館で未対応分野の探索方針を補う

### 適さない役割

- 一般的な書誌検索の主力 source
- `jp_lit_search` の既定横断対象
- 一次情報そのものの代替

---

## MCP と Skill の分担

### Skill 側

Skill は次を担当する。

- いつレファ協を見るべきか判断する
- どの query でレファ協を引くか決める
- レファ協の事例や調べ方マニュアルを読んで、次の source / query を決める
- レファ協を一次情報としてではなく、探索ルートの手がかりとして扱う

### MCP 側

MCP は次を担当する。

- レファ協 API から候補を検索・取得する
- タイトル、概要、回答プロセス、参考資料などを構造化して返す
- 書誌 DB とは別種の結果であることが分かるように返す

---

## 既存ツールへどう入れるべきか

### 入れ方として避けたいもの

- `jp_lit_search(source=crd)` をいきなり既存の書誌系 source と同じ感覚で追加する
- source 未指定横断検索に含める

これをやると、

- 文献候補
- 調べ方マニュアル
- レファレンス事例

が同列に並び、検索結果の意味がぶれやすい。

### 有力案

#### 案A: 専用ツールにする

例:

- `jp_lit_search_research_guides`
- `jp_lit_search_reference_cases`

利点:

- 何を返すツールかが分かりやすい
- 既存の書誌検索 UX を壊さない
- Skill からも呼び分けやすい

欠点:

- ツール数が増える

#### 案B: `jp_lit_search` に `source=crd_manual` / `source=crd_reference` を追加する

利点:

- 既存 API との見た目は揃う

欠点:

- 書誌 source と同列に見えやすい
- source の意味がかなり異質

現時点では **案A（専用ツール）** の方が自然。

---

## 出すならどのデータを先に扱うか

レファ協の4種類のうち、優先順位は次のとおり。

1. 調べ方マニュアル
2. レファレンス事例
3. 特別コレクション
4. 参加館プロファイル

理由:

- 調べ方マニュアルは `research_guide` と最も相性がよい
- レファレンス事例は「回答プロセス」が有益
- 特別コレクションは主題探索の補助になる
- 参加館プロファイルは補助的な意味が強い

---

## Skill から見た使いどころ

レファ協は次のような場面で使うのが自然。

- `何で調べればいいかわからない`
- 0件が続き、探索方針を見直したい
- 人文学・地域資料・郷土資料で、一般的な書誌 DB だけでは弱い
- 参考資料や索引類の使い方を知りたい
- 回答に至るプロセスを参考にしたい

逆に、次の場面では後回しでよい。

- 既に具体的な書名や論文名が分かっている
- まず所蔵館を確認したい
- デジコレ OCR のページ特定が目的

---

## 実装順の提案

### 今やらない理由

- 価値が低いからではない
- 置き場所を誤ると UX がぶれるから
- 先に README / Skill / 公開導線を整える方が優先度が高かった

### 次にやるなら

1. 調べ方マニュアルだけを対象にした専用ツールの設計
2. Skill の `research_guide` フローからの呼び出し設計
3. 既存書誌検索とは別系列で UI / 出力を整理

---

## 最終方針

- レファ協は有望
- ただし「書誌検索 source」としてではなく「調査支援 source」として扱う
- 既存 `jp_lit_search` へ雑に混ぜない
- 先に入れるなら `調べ方マニュアル` を対象とした専用ツールから始める
