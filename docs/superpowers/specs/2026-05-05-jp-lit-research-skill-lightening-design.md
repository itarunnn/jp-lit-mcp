# jp-lit-research Skill 軽量化設計

## 背景

`jp-lit-research` は日本語文献調査で頻繁に使う中核 Skill だが、現在の `SKILL.md` は約400行あり、調査哲学、DB選択、検索手順、キャッシュ操作、出力形式、禁止事項が同じ階層に並んでいる。そのため、実運用で重要な `調査ログ` や `本文確認状態` のような必須要件が長文内に埋もれやすい。

直近の文献調査では、Skill 内に「何件取得→何件抽出を出す」と書かれていたにもかかわらず、初回回答で調査経過の出力を落とした。これは個別ミスとして修正するだけでなく、Skill 構造そのものを「毎回守る要件が目に入る形」に変える必要がある。

## 目的

- `SKILL.md` を短い司令塔にする。
- 回答末尾の `調査ログ` を落としにくくする。
- `online=true` / PDFリンク / デジコレ入口と、本文読了を混同しない出力を安定させる。
- 詳細なDB選択・証拠評価・キャッシュ操作は参照ファイルへ分け、必要時だけ読む構造にする。
- 既存の文献調査能力は維持し、読み落としやすい長文指示を減らす。

## 非目的

- jp-lit MCP のツール仕様は変更しない。
- 新しい DB source や外部 API は追加しない。
- 文献調査の厳密性を下げない。
- 既存の workflow / heuristics / reference ファイルを大規模に再設計しない。
- `jp-lit-verification` Skill は今回の対象外とする。

## 設計方針

### 1. `SKILL.md` は薄いルーターにする

本体に残す内容は次に絞る。

- YAML frontmatter
- 起動条件と継続条件
- `必ず守る` チェックリスト
- 最小ワークフロー
- intent 分類の短表
- 参照ファイルの読み分け
- 標準出力テンプレート
- 最終回答前チェック

目標行数は 120〜180 行程度とする。行数そのものを絶対条件にはしないが、200行を超える場合は、何が本体に残る必要があるのか再レビューする。

### 2. `調査ログ` は本体で二重に強調する

`調査ログ` は今回の失敗から見て最重要の運用要件なので、参照ファイルだけに置かない。本体の冒頭チェックリストと標準出力テンプレートの両方に置く。

必須列は次の通り。

```md
| # | source | query | total | 取得件数 | 抽出件数 | 備考 |
```

`total` と `取得件数` は区別する。ページングや横断検索の制約がある場合は `備考` に書く。検索しなかった場合も、`調査ログ: 今回は検索なし（理由: ...）` と明示する。

最終回答前チェックにも `調査ログを書いたか` を入れる。候補リストを短縮する場合でも、調査ログだけは省略しない。

### 3. 本文確認ラベルは本体に残す

`availability.online=true` は本文読了ではない、というルールは本体に残す。詳細な証拠評価は `reference/03-evidence-and-output.md` に置くが、最低限の本文ラベルは本体のテンプレートに残す。

本文ラベルは次を標準とする。

- `本文: 未確認`
- `本文: オンライン入口あり未読`
- `本文: オンライン入口なし`
- `本文: アクセス制限`
- `本文: 確認済み`

### 4. 参照ファイルを役割別に読む

本体から次の参照先を明示する。

- `reference/01-core-workflow.md`: 探索ループ、計画提示、ユーザー確認
- `reference/02-source-and-query.md`: source 選択、検索語展開、ページング
- `reference/03-evidence-and-output.md`: 証拠評価、候補表示、本文確認ラベル、出力詳細
- `heuristics/source-selection.md`: DB選択で迷う場合
- `heuristics/query-expansion.md`: 表記ゆれ・旧字・異体字が重要な場合
- `heuristics/evidence-grading.md`: 文献価値や根拠の強さを評価する場合
- `heuristics/failure-modes.md`: 0件、ノイズ過多、本文未確認、所蔵確認不成立の失敗対処
- `workflows/*.md`: intent ごとの詳細手順が必要な場合

### 5. cache/refine/export は詳細へ逃がす

`jp_lit_refine_results`、cache 一覧、export、annotation の詳細は通常の初動調査では毎回読まなくてよい。本体では「既存結果の再整理を求められたら `reference/01-core-workflow.md` と関連 tool 説明を読む」とだけ示す。

### 6. 検索前確認の原則と継続調査の扱いを分ける

新規テーマでは検索前に短い調査方針を提示し、ユーザーの確認後に検索する。すでに同一セッションで文献DB調査が発火し、ユーザーが「同様に」「次は近代で」など継続指示を出した場合は、既定方針を1〜2文で宣言して検索を続けてよい。この場合も、最終回答では `調査ログ` を必ず出す。

## 標準出力テンプレート

本体に次のテンプレートを置く。

```md
## 検索概要

- 検索回数:
- 使用 source:
- 検索語:
- 取得件数:
- 表示件数:
- 選別基準:

## 主要候補

1. 書誌
   - 種別:
   - source:
   - 確認:
   - 本文:
   - 優先:
   - 根拠:
   - 仮整理:
   - 次:

## 保留・弱い除外候補

## 除外した代表例

## 今回の確認範囲

## 調査ログ

| # | source | query | total | 取得件数 | 抽出件数 | 備考 |
```

ユーザーが短い回答を求めた場合でも、`今回の確認範囲` と `調査ログ` は圧縮して残す。

## 最終回答前チェック

本体に次のチェックリストを置く。

```md
- `検索概要` があるか
- `今回の確認範囲` があるか
- `調査ログ` があるか
- 各検索の `total` / `取得件数` / `抽出件数` を分けたか
- `online=true` やリンク確認を `本文: 確認済み` と混同していないか
- 本文未読の候補を本文読解済みのように要約していないか
```

## レビュー計画

### ループ1: 構造レビュー

観点:

- 本体に残す必須事項が残っているか。
- 詳細ルールが本体に居座っていないか。
- 参照ファイルへの導線が一段で辿れるか。
- `SKILL.md` が単独でも最低限の調査回答を完結できるか。

想定修正:

- 本体に残す項目の追加・削除
- 参照先一覧の整理
- 似た指示の重複削除

### ループ2: 行動レビュー

観点:

- `調査ログ` を最終回答から落としにくいか。
- `total` / `取得件数` / `抽出件数` の区別が明確か。
- `online=true` と `本文: 確認済み` を混同しないか。
- 本文未確認の候補を、本文読解済みのように要約しない構造か。
- 「検索前に計画提示」の原則と、継続調査時の合理的な扱いが矛盾しないか。

想定修正:

- 最終回答前チェックの追加
- テンプレート列名の調整
- 継続調査時の例外説明追加

### ループ3: テストレビュー

観点:

- `tests/skillGuide.test.ts` が長文Skillの文言固定に依存しすぎていないか。
- 本体の必須契約をテストしているか。
- 詳細参照ファイルに残すべきルールをテストしているか。
- 将来の軽量化を妨げる brittle なテストになっていないか。

想定修正:

- `SKILL.md` の必須見出し・テンプレート・参照先だけを検証する。
- 詳細ルールは該当 reference / heuristics ファイル側で検証する。
- 「397行の本文に特定文言がある」系のテストは減らす。

### ループ4: 前向きテスト

実際の依頼に近いケースで、Skill構造が効くか確認する。

ケース:

- テーマ文献調査: 「近世の賭博史について文献DBで調べて」
- 所蔵確認: 「この2件はオンラインで読める？読めない場合は近隣所蔵館」
- 本文可否確認: 「オンライン本文がある日本語文献は？」

評価:

- `検索概要` がある。
- `今回の確認範囲` がある。
- `調査ログ` がある。
- 本文未読を本文読了として扱っていない。
- 取得件数と抽出件数が分かれる。

## 実装ステップ

1. `skills/jp-lit-research/SKILL.md` の新構成を作る。
2. `reference/01-core-workflow.md` / `reference/02-source-and-query.md` / `reference/03-evidence-and-output.md` を点検し、本体から移した詳細が失われていれば補う。
3. `tests/skillGuide.test.ts` を新しい契約に合わせて更新する。
4. `npm test -- tests/skillGuide.test.ts` を実行する。
5. 構造レビュー、行動レビュー、テストレビューを順に行い、必要な修正を入れる。
6. 前向きテストを小さな固定プロンプトで行う。外部検索を伴う実行が重い場合は、Skill本文だけを読ませる机上テストに切り替え、その制約を記録する。
7. `npm test` と `npm run build` を実行する。
8. `npm run skills:install` または既存のインストール手順で、配布元からローカルインストール済み Skill へ反映できることを確認する。

## 成功条件

- `SKILL.md` が短くなり、必須出力契約が冒頭で読める。
- `調査ログ` テンプレートが本体にある。
- `今回の確認範囲` テンプレートが本体にある。
- 本文確認ラベルが本体にある。
- 詳細ルールは参照ファイルに残っている。
- `tests/skillGuide.test.ts` が新構造に対応している。
- 全テストと build が通る。

## リスクと対策

- リスク: 軽量化で重要ルールを削りすぎる。
  - 対策: 必須ルールは本体、詳細は参照ファイルに残す。削除ではなく移動を基本にする。

- リスク: 参照ファイルを読まなくなり、詳細判断が粗くなる。
  - 対策: intent や状況ごとの「読むべき参照先」を本体に短く明記する。

- リスク: テストが文言固定になり、今後の改善を妨げる。
  - 対策: 見出し・テンプレート・参照先・必須ラベルの存在を検証し、長文説明の全文一致は避ける。

- リスク: 配布元とインストール済み Skill がずれる。
  - 対策: 配布元を唯一の編集対象にし、最後にインストール手順で同期する。
